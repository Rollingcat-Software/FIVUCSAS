## 1. Why
Today `docs.fivucsas.com` is **4 disconnected pages**: a landing `index.html`, three OpenAPI
reference pages (`/identity/`, `/biometric/`, `/sdk/`), and the (now-linked) `diagrams.html`
gallery. Meanwhile the `docs/` submodule holds a rich, chaptered body of real content
(`00-meta`, `01-getting-started`, `02-architecture`, `03-development`, `plans`,
`presentations`, 8 ADRs) that **never reaches the web**. There is no narrative, no
left-nav, no search, and the 33 architecture diagrams live apart from the prose that
explains them.

Goal: a single navigable **book** — chapters of prose with **diagrams inline as figures
next to the text that explains them**, plus the existing visual gallery as an appendix —
with search, dark mode, mobile, and EN/TR, sourced from the Markdown we already maintain.

## 2. Tooling decision

| Option | Verdict |
|---|---|
| **VitePress** ✅ **(recommended)** | Vue-based SSG, first-class **Mermaid** via `vitepress-plugin-mermaid` (renders our diagrams from the same fenced source), built-in local search, dark mode, responsive sidebar/TOC, i18n (EN/TR) out of the box, MIT, tiny output. Fits our existing **Docker** docs container (swap nginx-static for a built static site — still static, still Dockerized, no new infra). |
| Docusaurus | Also excellent (React, Algolia/local search, versioning). Heavier than we need; we have no React-docs requirement. Fallback if we later want doc versioning. |
| mdBook / MkDocs-Material | Great books, but Mermaid support is bolt-on and theming/i18n is less smooth than VitePress for our EN/TR + brand. |
| Keep hand-built static | Lowest effort, but we'd re-implement nav/search/figure-flow by hand and keep prose off the web. Not worth it. |

**Pick VitePress.** It renders Mermaid inline (so chapters carry their own figures), keeps
the build static (no runtime, deploys behind the same Traefik/Docker setup), and reuses our
Markdown verbatim.

## 4. How diagrams integrate
- **Inline figures:** every chapter embeds the relevant Mermaid fence directly, rendered by
  `vitepress-plugin-mermaid`. The diagram lives *next to its explanation* (the thing the
  current standalone gallery can't do).
- **Hero SVG:** the hand-built C4 container SVG (now the first item in `diagrams.html`) becomes
  the opening figure of Ch. 4 — embedded as a static `.svg` (crisp, zoomable, theme-neutral).
- **Gallery appendix:** keep the current `diagrams.html` verbatim as Appendix A (the "see every
  diagram at once" view). Zero rework; it already has zoom/pan/TOC.
- **Single source of truth:** chapter fences and the gallery should pull from the same `.md`
  diagram source so they never drift (extract the 33 fenced blocks into `docs/diagrams/*.mmd`
  partials, included by both). The 12 legacy `.drawio` files stay as an editable archive only.

## 5. Migration plan (phased, each phase shippable)
1. **Scaffold** VitePress in `docs-site/book/` (config, theme, EN/TR, brand colors, Mermaid plugin). Keep the existing static pages live in parallel.
2. **Wire OpenAPI:** embed the three Swagger/Redoc pages as-is under Part VI (iframe or `vitepress-openapi`).
3. **Port prose:** symlink/import the `docs/` chapters; fix the 3 known stale spots (NGINX→Traefik, pre-V55 ER, Fall-2025 progress) flagged in `ARCHITECTURE_DIAGRAMS.md §11`.
4. **Inline diagrams:** move the 33 fenced blocks into per-chapter figures; keep the gallery as Appendix A.
5. **Cut over:** point the `fivucsas-docs` container build at the VitePress output; 301 old paths; submit sitemap.
6. **Author the 2 new high-value chapters** (Threat model, KVKK lifecycle) — see §6.
