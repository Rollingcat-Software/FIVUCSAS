## USER-BUG-1: Face enrollment proceeds without a detected face

**User words (TR → EN):**
> Tavanı göstersem bile devam ediyor yüz görmeden. Ama db kaydı başarısız dedi. O noktada kontrol ediyor sanki.
>
> *Even when I aim the camera at the ceiling, [enrollment] continues without seeing a face. But it said DB save failed. It's like it's only checking at THAT point.*

**Symptom:** the enrollment UI does not gate progression on real-time face detection. Auto-capture / advance fires regardless of whether a face is in view; failure surfaces only when the server rejects the empty/poor descriptor at persistence time.

**Why it matters:** every prospect who flies through enrollment in poor lighting hits a confusing "kayıt başarısız" at the very end with no actionable feedback during capture. Combined with PERF #1 (slow JVM heap → slow enroll endpoint), this looks like server flakiness when it is actually a missing front-end gate.

**Suspected root cause:** the `faceDetected` / `passiveLivenessOk` signals from the local detector are computed but not used to block the auto-capture timer or the "Continue" button. The fix is a state-machine guard, not a new detector.

**Files to inspect (web-app):**
- `src/features/auth/components/FaceCaptureStep.tsx`
- `src/features/auth/components/FaceEnrollmentFlow.tsx`
- `src/features/auth/hooks/useFaceChallenge.ts`

**Fix shape:**
- Refuse to advance / auto-capture when `faceDetected !== true || passiveLivenessOk !== true`.
- Localized inline message + disabled CTA.
- Defense-in-depth: bio `/v1/face/enroll` returns 422 `{"error":"NO_FACE_DETECTED"}` BEFORE pgvector touch.
- Vitest unit test on `useFaceChallenge` driving `faceDetected: false` to assert no advance.

**Tracking:** Task #28 in TaskList. Branch `fix/face-enrollment-no-face-gate`.

## USER-BUG-2: YOLO card-type detection picks the wrong type

**User words:** "Yanlış buluyor" — *finds the wrong [class].*

**Context:** this is the client-side YOLO step in NFC-document enrollment that classifies the card image (eski kimlik / yeni kimlik / ehliyet / pasaport) before the user is prompted to hold it to the NFC chip.

**Suspected root causes (in priority order):**
1. Label-order mismatch between training and runtime (YOLO exported with one class order, JS reads alphabetical).
2. Preprocessing mismatch (BGR/RGB, letterbox, normalization).
3. No confidence threshold; argmax-of-noise reported as a confident answer.
4. Stale model artifact — `manifest.json` SHA256 doesn't match what front-end loaded.
5. Worst case: model itself is undertrained for the TR ID variants.

**Defensive fix even if model is the root cause:** ≥0.6 confidence floor, fall back to "Kart türü algılanamadı / Card type unrecognized — try better lighting" with a manual-select UI.

**Tracking:** Task #29. Branch `fix/card-type-yolo-classification` (web-app likely; bio `card_type_router` is slated for removal in master roadmap Phase 6.4).

## USER-BUG-3: Math + shape-drawing biometric puzzles don't work

**User words:** "Bu matematik ve şekil çizme kısımları çalışmıyor" — *the math and shape-drawing parts don't work.*

**Context:** biometric-puzzles page (split from auth-methods-testing on 2026-04-24) hosts 23 micro-challenges including FacePuzzle (mesh overlay, 91b1b6e), HandGesturePuzzle (skeleton), MathPuzzle*, ShapeDrawingPuzzle*. PuzzleRunnerModal was wrapped in React.lazy() per master roadmap P6.10.

**Likely failure modes (from review experience):**

**Tracking:** Task #30. Branch `fix/biometric-puzzles-math-shape`.

## USER-BUG-4: SMS OTP enrollment fails with correct code

**User words (2026-05-01):**
> sms otp enrollmend fails even if incoming code in sms is entered correctly.

**Symptom:** user receives a 6-digit code via Twilio, types it correctly, server rejects.

**Likely root causes (rank by probability):**
1. String normalization mismatch (zero-width / bidi markers / Arabic-Indic numerals from Turkish keyboards; `.equals()` without `.trim()` + Unicode-NFC).
2. Hash asymmetry (hash on send + plain-compare on verify).
3. Wrong session/method binding (mfa_sessions row found by tenant X, verified for tenant Y).
4. Replay-guard `consumed_at` set on emit instead of on verify-success.
5. Front-end field-name mismatch (`code` vs `otp` vs `verificationCode`).
6. Cooldown off-by-one.

**Tracking:** Task #32. Branch `fix/sms-otp-enrollment-mismatch`. Investigation requires reading SmsOtpAuthHandler.java + SmsOtpStep.tsx + audit_logs for actual MFA_STEP_FAILED reason.
